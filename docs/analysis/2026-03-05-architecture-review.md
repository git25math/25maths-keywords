# 架构评审 v1: 扩展计划可行性分析

> **日期**: 2026-03-05
> **版本**: v1.0
> **目的**: 对照实际代码架构，批判性评估四层扩展计划的可行性

## 架构现状快照

| 维度 | 现状 | 量化 |
|------|------|------|
| 模块系统 | 无（19 个 `<script>` 全局变量） | ~180-200 全局变量 |
| 数据层 | levels.js 全量加载 | 280KB / 4,128 行 |
| 渲染层 | 16 Panel 全量预渲染 | 无懒加载 |
| 外部依赖 | 仅 Supabase JS CDN | 1 个依赖 |
| 数学渲染 | 无 | 无 KaTeX / MathJax |
| 离线方案 | build-single.py 单文件 | 无 Service Worker |
| 数据同步 | localStorage + Supabase | 2s debounce sync |

## 五大架构硬约束

### 1. getPairs() 强耦合
**影响范围**：所有 7 个学习模式
**现状**：`getPairs(li)` 返回 `{ word, def, lid }` 三元组，所有模式的渲染、交互、评分都绑定此结构
**约束**：要加公式 / 图示 / 例题字段，必须扩展数据结构或新建独立管道
**影响评估**：Layer 2 实现成本从"低"上调为"中"

### 2. 无 LaTeX/数学渲染
**影响范围**：Layer 2（公式卡）、Layer 3（练习题）、Layer 4（模拟考试）
**现状**：项目唯一外部依赖是 Supabase JS，无任何数学渲染库
**约束**：CIE 题库 2,300 题 + 25maths 网站 1,200 题全部使用 LaTeX 格式
**影响评估**：需引入 KaTeX（~300KB gzip ~90KB），是当前总依赖量的数倍

### 3. levels.js 数据瓶颈
**影响范围**：首屏加载性能、移动端体验
**现状**：280KB / 4,128 行，存储 2,200 词的完整数据
**约束**：加入公式/例题/图示数据预估膨胀到 500KB+
**影响评估**：必须先拆分为按 board 按需加载的 JSON 文件

### 4. 全局变量命名空间
**影响范围**：可维护性、新功能开发效率
**现状**：~180-200 个全局变量，19 个文件无模块隔离
**约束**：再加 3-4 个功能模块容易命名冲突，调试困难
**影响评估**：虽不阻塞短期开发，但中期必须收拢为命名空间对象

### 5. Panel 全量预渲染
**影响范围**：DOM 初始化性能、内存占用
**现状**：16 个 Panel 在 index.html 中全部预渲染
**约束**：每新增一个功能模块 = 新增一个 Panel = 增加 DOM 和 JS 初始化开销
**影响评估**：短期可接受（<20 Panel），但 Layer 3-4 需要懒加载机制

## 逐层可行性修正

### Layer 1: 词汇层 ✅ 已完善，无需改动

### Layer 2: 概念层 ⚠️ 成本上调

| 原评估 | 修正评估 | 原因 |
|---------|----------|------|
| 实现成本：低 | 实现成本：中 | getPairs() 耦合 + 需引入 KaTeX |

**推荐方案**：新建独立 concept 数据结构和 Panel（方案 B），避免改动现有模式渲染逻辑。KaTeX 按需加载（仅概念 Panel 首次打开时加载）。

### Layer 3: 练习层 ⚠️ 有资产但集成有门槛

**利好**：
- 25maths-website: ~1,200 题（202 JSON 文件，标准选择题格式）
- CIE pipeline: ~2,300 题（含 subtopic/difficulty/cognitive level）

**门槛**：
- 两个数据源都依赖 LaTeX（前置依赖 KaTeX）
- 数据格式与 levels.js 完全不同，需独立数据加载管道
- 数据量大（估计 2-5MB），必须按需加载

**可行路径**：先做 25maths-website 选择题（格式已标准化），CIE 真题第二阶段。

### Layer 4: 评估层 ❌ 当前架构无法直接支持

- 需要计时器 + 多题型混合 + 评分 rubric
- 需要服务端评分（Edge Function 或外部 API）
- 全局变量 + 无模块系统下复杂度不可控
- **建议**：推迟到架构升级后，或作为独立子项目

## 扩展计划遗漏项

1. **缺少"架构升级阶段"** — 在 200 个全局变量上堆功能不可持续
2. **缺少 LaTeX 渲染策略** — Layer 2-4 的共同前置依赖
3. **缺少数据管道设计** — 练习题 / 真题的导入/转换/存储方案未定义
4. **缺少离线策略** — 数据量膨胀后 build-single.py 不再适用
5. **缺少性能预算** — 280KB→500KB+ 对移动端首屏的影响未量化

## 修正后分阶段路线

```
Phase 0: 架构准备（1-2 周）
├── levels.js 拆分为按需加载 JSON
├── KaTeX 按需加载管道
└── 全局变量收拢为命名空间对象（可选）

Phase 1: 概念卡 — 方案 B（1 周）
├── 独立 concept 数据结构 + Panel
├── 公式渲染（KaTeX）
└── 与词卡交叉链接

Phase 2: 选择题练习（1-2 周）
├── 导入 25maths-website 1,200 题
├── Quiz 模式扩展支持数学题
└── 按 topic/difficulty 筛选

Phase 3: 教师工具增强（1 周）
├── 作业关联练习题
├── 班级成绩分析增强
└── 错题分析报告

Phase 4: 评估层（需重新评估）
├── 取决于 Phase 0-2 架构成熟度
└── 可能需要独立子项目
```

## 核心结论

扩展计划**方向正确，但低估了架构改造成本**。当前架构为"词汇学习工具"优化，要变成"学习支撑平台"需先打地基（Phase 0）。最有价值的快速突破点是 **25maths-website 的 1,200 道现成练习题**——数据已有，只差渲染和集成。
